## 组件

我们可以考虑多层次的调度。我们可能会被要求设计由组织内部完成的调度，以便在他们自己的机器集群上运行任务。在那里，他们必须找到充足的资源，并且需要决定首先运行哪个任务。

另一方面，我们也可能被要求设计云提供商用来安排来自多个客户端的任务的调度。云提供商需要决定首先运行哪个任务以及首先处理哪个客户端，以在不同租户之间提供适当的隔离。

所以，总的来说，我们系统的主要组成部分是：

- **客户：**他们启动任务执行。
- **资源：**任务在这些组件上执行。
- **调度器**：调度器执行客户端和资源之间的进程，并决定哪个任务应该首先获得资源。

调度程序将任务放入队列以进行资源分配

如上图所示，需要将传入的任务放入队列**中**。这是因为以下原因：

- 我们现在可能没有足够的可用资源。
- 存在任务依赖性，一些任务需要等待其他任务。
- 我们需要将客户端与任务执行分离，以便他们可以将工作交给我们的系统。然后我们的系统将其排队等待执行。

让我们设计一个任务调度系统，它应该能够调度任何任务。通常，许多任务的生命周期相对较短——从几秒到几分钟。对于长时间运行的任务，我们可能需要在应用程序级别进行定期校验和恢复的能力，以从可能的故障中恢复。

假设我们机队中的某个服务器可以满足每个任务的计算需求。对于需要很多服务器的任务，应用程序需要将它们分解为我们系统的更小任务，或者从集群管理器中获取长期资源。

## 设计

当任务需要调度时，它应该包含以下信息：

- **资源要求：**要求包括它需要多少 CPU 内核、执行此任务需要多少 RAM、需要多少磁盘空间、磁盘访问速率应该是多少（每秒输入/输出速率，或 IOPS），以及任务执行需要多少个 TCP 端口，等等。但是，客户很难量化这些要求。为了纠正这种情况，我们提供了不同级别的资源，例如基本资源、常规资源和高级资源。客户可以根据这些层指定要求。

- 依赖性：

  从广义上讲，任务可以分为两种类型：依赖性和独立性。

  - **从属任务**需要执行一项或多项附加任务才能完整执行。这些任务必须按顺序运行。对于依赖任务，客户端应提供给定任务所依赖的任务列表。
  - **独立任务**不依赖于任何其他任务的执行。独立任务可以并行运行。我们应该知道一个任务是依赖的还是独立的。依赖信息有助于按顺序执行依赖任务和并行执行独立任务，以有效利用资源。

- **客户**：云提供商的客户是希望执行其任务的个人或组织，从小型企业到大型企业。

- **速率限制器**：客户可用的资源取决于他们支付的成本。为了我们服务的可靠性，限制任务数量很重要。例如，�*X*每小时允许进入系统的任务数。其他人会收到类似“超出限制”的消息，而不是接受任务并延迟响应。速率限制器限制客户端根据其订阅安排的任务数量。如果超过限制，它会向客户端返回一条错误消息，指出已超过速率限制。

- **任务提交者**：如果任务成功通过限速器，则任务提交者承认该任务。没有一个任务提交者。相反，我们有一个节点集群，可以接受越来越多的任务。

- **Unique ID generator**：它为新接受的任务分配唯一的ID。

- 数据库

  ：任务提交者所接的所有任务都存储在一个分布式数据库中。对于每个任务，我们都有一些属性，除了一个属性之外的所有属性都存储在关系数据库中。

  - **关系数据库（RDB）**：关系数据库存储任务ID、用户ID、所需资源、执行上限、客户端尝试的总数、延迟容忍度等，如下表所示。[我们可以在此处](https://www.educative.io/collection/page/10370001/4941429335392256/6521598450597888)找到有关 RDB 的详细信息。
  - **图数据库（GDB）**：这是一种非关系型数据库，使用图数据结构来存储数据。我们用它来构建和存储相关任务的有向无环图 (DAG)，由任务提交者进行拓扑排序，以便我们可以根据该 DAG 调度任务。[我们可以在此处](https://www.educative.io/collection/page/10370001/4941429335392256/6521598450597888)找到图形数据库的更多详细信息。

指向思考

###### 问题

为什么我们将任务存储在数据库中？为什么我们不应该将任务直接推送到队列中？

隐藏答案

队列不会永久保存数据。我们为使用的排队服务付费。所以，我们只是将那些即将执行的任务推送到队列中。成功执行的任务需要从队列中移除。此外，还有不同的调度类型。任务可以安排一次、每天、每周、每月或每年。因此，我们必须将任务保存在我们存储的某个地方。

- **分布式队列**：它由一个队列和一个队列管理器组成。队列管理器添加、更新或删除队列中的任务。它跟踪我们使用的队列类型。它还负责将任务保持在队列中，直到它成功执行。如果任务执行失败，该任务将再次在队列中可见。队列管理器知道在高峰时间运行哪个队列以及在非高峰时间运行哪个队列。
- **队列管理器**：如果任务执行成功，队列管理器会从队列中删除任务。如果先前执行失败，它还会使任务可见。如果执行失败，它会重试任务允许的尝试次数。
- **资源管理器**：资源管理器知道哪些资源是空闲的。它从分布式队列中提取任务并为它们分配资源。资源管理器跟踪每个任务的执行并将它们的状态发送回队列管理器。如果任务超出其承诺或要求的资源使用，则该任务将被终止，并将状态发送回任务提交者，任务提交者将通过错误消息通知客户端任务终止。
- **监控服务**：负责检查资源管理器和资源的健康状况。如果某些资源出现故障，它会提醒管理员修复资源或在需要时添加新资源。如果资源未被使用，它会提醒管理员删除它们或关闭它们。这里[详细讨论](https://www.educative.io/collection/page/10370001/4941429335392256/6310983387840512)监控服务的设计。

## 任务提交者

正如我们在上面看到的，我们在分布式任务调度器的设计中使用的每个组件都是分布式的，因此可扩展和可用。但是，任务提交者可能是单点故障。因此，为了处理这个问题，我们使用了一个节点集群。每个节点都必须承认任务，将任务发送给唯一的 ID 生成器进行 ID 分配，然后将任务连同任务 ID 一起存储在分布式数据库中。

有一个集群管理器，每个节点都会向其发送一个心跳，指示该节点是否正常工作。每个节点都会更新集群管理器有关已接纳任务的信息。集群管理器维护一个任务列表和接受该任务的节点 ID。如果一个节点无法执行任务，集群管理器将该任务移交给集群中的另一个节点。集群管理器本身被复制。
